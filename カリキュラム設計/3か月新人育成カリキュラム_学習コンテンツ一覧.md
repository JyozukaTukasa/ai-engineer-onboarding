# 3か月新人育成カリキュラム 学習コンテンツ一覧

## 位置づけ

この資料は、3か月の新人育成カリキュラムを設計する前段として、学習させるべきコンテンツを洗い出した草案です。  
AIを活用しながらも、1人で社外案件に参画できる水準を目指す前提で整理しています。

## 全体方針

| 領域 | 狙い | 到達イメージ |
| --- | --- | --- |
| 対顧客コミュニケーション | 顧客と信頼関係を築き、認識齟齬なく会話しつつ追加提案までできるようにする | ヒアリング、報連相、議事録、説明に加え、改善提案の会話ができる |
| 上流工程 | 要件を整理し、実装方針に落とし込めるようにする | 要件定義書、基本設計の素案を作れる |
| 下流工程 | 実装・テスト・運用の基本を押さえ、手を動かせるようにする | 小規模機能を自走して開発し、レビューを受けられる |
| AI活用 | AIを補助輪ではなく生産性向上手段として使えるようにする | 調査、設計、実装、テストでAIを適切に使い分けられる |

## 1. 対顧客コミュニケーション

| 項目 | 学ばせる内容 | 学習の観点 | 想定アウトプット |
| --- | --- | --- | --- |
| ビジネスマナー基礎 | あいさつ、言葉遣い、レスポンス速度、会議参加姿勢 | 信頼を損なわない基本動作を身につける | ロールプレイ評価 |
| 報連相 | 相談のタイミング、結論先出し、事実と推測の分離、エスカレーション | 問題を抱え込まずに共有できるようにする | 報告テンプレート記入 |
| テキストコミュニケーション | Slack/メールの書き方、依頼文、確認文、謝罪文、期限共有 | 非対面でも誤解なくやり取りできるようにする | メッセージ作成演習 |
| 会議コミュニケーション | 発言の仕方、質問の仕方、相手の意図確認、論点整理 | 会議で受け身にならず、理解を深める | 模擬会議参加 |
| ヒアリング | 目的確認、現状確認、課題抽出、制約条件の確認 | 要件定義に必要な情報を引き出せるようにする | ヒアリングシート作成 |
| 議事録作成 | 決定事項、宿題、論点、次回アクションの整理 | 会議内容を再利用可能な形で残す | 議事録作成演習 |
| 説明力 | 専門用語のかみ砕き、結論から話す、図解で補う | 非エンジニア相手にも説明できるようにする | 口頭説明演習 |
| 顧客折衝の基本 | 要望の受け止め方、優先順位調整、できること/できないことの伝え方 | 相手に配慮しつつ調整できるようにする | ケーススタディ回答 |
| 課題共有 | リスクの伝え方、代替案提示、期限影響の説明 | 問題発生時に信頼を落とさず対話できるようにする | 障害報告演習 |
| 提案型コミュニケーション | 顧客課題の深掘り、潜在ニーズの発見、改善案の提示、追加施策の会話 | 言われたことだけでなく、価値を広げる提案ができるようにする | 改善提案ロールプレイ |
| 案件拡張の視点 | 追加要望の拾い方、周辺課題の発見、体制拡張の会話、次フェーズ提案の入口 | 追加人材や追加案件につながる会話の種を作れるようにする | 追加提案メモ |
| 提案資料の基礎 | 課題、原因、改善案、期待効果、進め方の整理 | 提案を口頭だけでなく資料でも伝えられるようにする | 1枚提案資料 |

## 2. 上流工程

業務の流れに沿って学べるよう、実務での進行順に並べています。  
最初に全体像を押さえたうえで、理解→ヒアリング→整理→要件化→合意→設計へ進む構成です。

| 項目 | 学ばせる内容 | 学習の観点 | 想定アウトプット |
| --- | --- | --- | --- |
| システム開発の全体像 | 開発工程、関係者、成果物、役割分担 | 上流工程が何のためにあるか理解する | 工程整理シート |
| 要件定義の目的 | 要件定義とは何か、なぜ必要か、設計との違い | ただ書類を作るのではなく合意形成と理解する | 要件定義の説明資料 |
| 業務理解 | 業務フロー、利用者、現場課題、現状運用 | 顧客業務を理解せずに要件を決めない姿勢を身につける | 業務フロー図 |
| 要件ヒアリング | 目的、背景、利用者、機能、非機能、制約、優先順位の聞き出し | 要件定義に必要な情報を漏れなく集める | ヒアリング項目一覧 |
| 要件整理 | 要望と要件の違い、論点の分類、曖昧表現の具体化 | 断片情報を構造化してまとめられるようにする | 要件整理メモ |
| 制約条件整理 | 納期、予算、体制、既存システム、利用技術、法規制 | 現実条件の中で設計する視点を持つ | 制約整理表 |
| 優先順位付け | Must/Should/Could、段階リリース、MVP発想 | すべてを一度にやろうとしない判断を学ぶ | 優先順位マトリクス |
| 業務要件 | 対象業務、運用ルール、例外ケース、承認フロー | 現場運用を踏まえた要件化を学ぶ | 業務要件一覧 |
| 機能要件 | 画面、入力、出力、権限、通知、検索、帳票など | システムで実現すべきことを具体化する | 機能要件一覧 |
| 非機能要件 | 性能、可用性、拡張性、保守性、セキュリティ、監視、バックアップ | 見落とされやすい観点を最初から意識する | 非機能要件表 |
| 技術選定 | 言語、FW、DB、インフラ、外部サービスの比較観点 | 技術を好みではなく要件で選ぶ習慣をつける | 技術選定比較表 |
| 技術選定の判断軸 | 学習コスト、保守性、拡張性、採用実績、運用負荷、セキュリティ | 選定理由を説明できるようにする | 選定理由メモ |
| 要件定義書作成 | 章立て、記載粒度、前提条件、対象範囲、除外範囲 | 相手が読める要件定義書を作れるようにする | 要件定義書ドラフト |
| レビューと合意形成 | 認識齟齬の洗い出し、レビューコメントの扱い、承認取得 | 書いて終わりではなく、合意して初めて完了と理解する | レビュー反映版 |
| 基本設計 | システム構成、画面一覧、機能一覧、データの流れ、権限設計 | 実装前にシステムの骨格を決める | 基本設計書の一部作成 |
| 詳細設計 | API仕様、テーブル定義、バリデーション、例外処理、画面遷移 | 実装者が迷わないレベルまで具体化する | 詳細設計書サンプル |
| テスト設計の基礎 | 観点洗い出し、正常系/異常系、受入条件との対応 | 上流で品質を作り込む考え方を持つ | テスト観点表 |

## 3. 下流工程

業務の流れに沿って、準備→読む→作る→つなぐ→試す→直す→残す、の順で整理しています。  
AI駆動で働く前提でも、生成結果の妥当性を判断し、修正指示を出せる理解までは必須とします。

| 項目 | 学ばせる内容 | 学習の観点 | 想定アウトプット |
| --- | --- | --- | --- |
| 開発環境構築 | エディタ、ターミナル、パッケージ管理、ローカル実行 | 自力で開発開始できる状態を作る | 環境構築手順書 |
| プログラミング基礎（必須） | 変数、代入、条件分岐、繰り返し、関数、配列、オブジェクト、真偽値、比較、引数、戻り値 | AIが出したコードを読んで、何をしているか説明できるレベルまで固める | 基本文法課題 |
| プログラミング基礎（必須ではないが補助で触れる） | 型の厳密な分類、参照/値の細かな違い、スコープの深い仕様、クラス設計の詳細、アルゴリズム理論の深掘り | 未経験3か月では暗記より、実務で困らない理解を優先する | 補足メモ |
| 初歩ロジック整理 | やりたい処理を順番に分解する、条件で分ける、繰り返しを見つける、入力と出力を言語化する | いきなり設計書を書くのではなく、処理の流れを言葉で整理できるようにする | 処理手順メモ |
| Git基礎 | `add`、`commit`、`push`、`pull` を中心に、変更を保存して戻れる感覚をつける | 最初はチーム開発より、自分の作業を安全に管理する道具として理解する | Git演習 |
| Git実践 | ブランチ運用、PR、レビュー対応、コンフリクト解消 | チーム開発の流れに慣れる | 模擬PR提出 |
| フロントエンド基礎 | HTML、CSS、JavaScript、DOM、イベント | 画面の基本構造を理解する | 静的ページ制作 |
| フロントエンド実践 | コンポーネント設計、状態管理、フォーム、API連携 | 業務画面を作る基礎力をつける | CRUD画面実装 |
| バックエンド基礎 | HTTP、REST、ルーティング、バリデーション、例外処理 | APIの役割と責務を理解する | API作成課題 |
| バックエンド実践 | 認証、認可、ログ、設定管理、エラーハンドリング | 実務で必要な基本設計を押さえる | 簡易業務API実装 |
| データベース基礎 | テーブル、主キー、外部キー、正規化、SQL | データ設計と問い合わせの基礎を学ぶ | SQL演習 |
| データベース実践 | CRUD、インデックス、トランザクション、マイグレーション | 実運用を意識したデータ操作を学ぶ | テーブル設計課題 |
| インフラ基礎 | OS、ネットワーク、DNS、Webサーバ、クラウド概念 | アプリが動く土台を理解する | 構成図読み解き |
| インフラ実践 | デプロイ、環境変数、ログ確認、監視の入口 | 作って終わりではなく動かす視点を持つ | デプロイ演習 |
| セキュリティ基礎 | 脆弱性の基本、認証情報管理、権限制御、入力値検証 | 最低限の事故を防ぐ意識を持つ | 脆弱性確認チェック |
| テスト実装 | 単体テスト、結合観点、テストデータ、再現確認 | 品質をコードで担保する入口を学ぶ | テストコード作成 |
| デバッグ | ログ、ブレークポイント、原因切り分け、再現手順 | 詰まった時に自走できる力を養う | バグ調査レポート |
| ドキュメント化 | README、セットアップ手順、設計補足、運用メモ | 他者が引き継げる形で残す習慣をつける | README作成 |

### 下流工程におけるAI駆動エンジニアの最低理解ライン

| 観点 | 必須レベル | 補足 |
| --- | --- | --- |
| コード読解 | `if`、`for`、関数、配列操作、API呼び出しを読んで説明できる | AI生成コードのレビューに直結する |
| 修正指示 | 「どこをどう変えるか」を文章でAIに指示できる | 自分で仕様差分を認識できることが前提 |
| エラー対応 | エラーメッセージを読み、原因候補を3つ程度に絞れる | 丸投げせず切り分けできる必要がある |
| 画面/API/DBの関係理解 | 画面入力がAPIを通ってDBに保存される流れを説明できる | フロント、バック、DBを分断して学ばないため |
| 検証責任 | AIの出力をそのまま出さず、動作確認・レビュー・修正ができる | AI活用でも最終責任は人が持つ |

## 4. AI活用

| 項目 | 学ばせる内容 | 学習の観点 | 想定アウトプット |
| --- | --- | --- | --- |
| AI活用の前提 | AIでできること、できないこと、過信しない姿勢 | AIを盲信せず、補助として扱う | 活用ルール整理 |
| プロンプト基礎 | 目的、前提、制約、出力形式の伝え方 | 欲しい回答を引き出すための指示力を身につける | プロンプト改善演習 |
| 調査への活用 | 技術調査、比較、要点整理、初期理解 | 学習速度を上げる活用方法を学ぶ | 調査メモ |
| 設計への活用 | 要件整理、観点出し、レビュー観点生成 | 思考の補助として使えるようにする | 設計たたき台 |
| 実装への活用 | 雛形生成、エラー原因仮説、リファクタ補助 | コード生成を鵜呑みにしない姿勢を持つ | AI利用実装ログ |
| テストへの活用 | テスト観点洗い出し、ケース案生成、異常系整理 | 見落としを減らす支援として使う | テストケース案 |
| レビューへの活用 | コードレビュー観点、設計レビュー観点、文書校正 | セルフレビューを強化する | レビュー結果メモ |
| AI利用時の注意 | 情報漏えい、機密情報、誤回答、著作権、検証責任 | 実務で安全に使うためのルールを守る | 利用ガイド理解確認 |

## 5. カリキュラム設計時に意識したい横断テーマ

| テーマ | 含めたい内容 | 狙い |
| --- | --- | --- |
| 自走力 | 調べ方、質問の仕方、詰まり方の共有 | 放置せず前進する行動を身につける |
| 品質意識 | 再現確認、レビュー、テスト、ドキュメント化 | 雑に終わらせない習慣を作る |
| 仕事の進め方 | タスク分解、見積もり、優先順位、進捗共有 | 実務に近い働き方を体験させる |
| 顧客視点 | 相手の目的、業務理解、期待値調整 | 作ることより価値提供を意識させる |
| 提案視点 | 追加改善案、次フェーズ案、体制拡張の可能性 | 今の案件を次の機会につなげる視点を持たせる |
| AIと人の役割分担 | AIに任せる部分、本人が理解すべき部分 | AI時代でも説明責任を持てる人材にする |
